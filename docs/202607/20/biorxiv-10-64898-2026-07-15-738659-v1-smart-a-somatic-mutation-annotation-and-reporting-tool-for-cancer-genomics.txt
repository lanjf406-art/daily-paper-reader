Because somatic variants frequently map to multiple 179 overlapping transcripts with divergent protein-level consequences, the selection of 180 which transcript to query against curated knowledgebases (and which to display in the 181 final output) has direct implications for the output oncogenicity classification, therapeutic 182 evidence level, and prognostic associations assigned to a given variant (demonstrated 183 empirically in Section 4.5.2). # 4 Implementation 217 218 4.1 OncoKB API integration and VCF reintegration 219 Although OncoKB annotations are publicly accessible, automated large-scale 220 integration requires the use of the OncoKB REST API, as the framework does not 221 natively support VCF-formatted inputs. SMART addresses this limitation through an 222 intermediate transformation layer that extracts variant-level features from the VEP-223 annotated VCF and reformats them into the per-variant input expected by the OncoKB 224 API. 225 For each variant, the prioritised HGVS.p notation (Section 3.1) and gene identifier are 226 submitted to the API to retrieve curated annotations, including oncogenicity 227 classification, biological mutation effect, the full hierarchy of curated evidence levels 228 (therapeutic sensitivity L1–4, resistance R1–R3, diagnostic Dx1–3, prognostic Px1–3, 229 FDA regulatory), and tumour-type–specific drug associations. API responses are 230 returned in structured JSON, parsed, and normalised into a flat set of fields that are 231 reintegrated into the original VCF as additional INFO entries. The resulting VCF 232 therefore consolidates VEP consequence-level annotations and OncoKB curated 233 knowledge within a single, standard-compliant representation, preserving compatibility 234 with all downstream tools that accept VCF input. To our knowledge, this direct 235 integration of API-derived OncoKB annotations into a VCF-based annotation framework 236 has not been previously described in the literature. 237 238 Variants are routed to the appropriate OncoKB endpoint based on type. SNVs, indels, 239 MantaINS (structural insertions) and MantaBND (translocation breakends) are 240

## 4.2 Docker container and reference data installer 266 All software components (VEP and its plugins, GATK, the OncoKB client, the table-267 generation scripts, and their dependencies) are distributed within a single Docker 268 container. Dependencies are version-locked at image build time, providing a 269 reproducible execution environment and removing the need for users to manage 270 individual tool installations. The container is published with a fixed tag corresponding to 271 the manuscript version (smart:vX.Y) and is accompanied by a configuration file 272 (Config.yaml) that exposes all runtime parameters, including reference paths, the 273 transcript whitelist, OncoKB token, and the per-tier annotation field selection used to 274 compose the output files. 275 276 Reference data setup is managed by a dedicated installer script (utils/get_ref_files.sh), 277 which downloads and prepares all required annotation resources in a single command: 278 the GRCh38 reference genome and UCSC liftover chain, ClinVar, CIViC, REVEL, 279 gnomAD constraint metrics (LOEUF), Cancer Hotspots (linear and 3-D), and the 280 Ensembl VEP cache and associated plugins. The installer is idempotent, meaning 281 completed steps are detected and skipped on subsequent runs and records database 282 versions and download dates in an auto-generated configuration file, ensuring that the 283 exact reference state used for any analysis can be reconstructed. # 5 Discussion 463 SMART addresses a practical gap between raw somatic variant annotation and the 464 structured outputs needed in routine research genomics workflows. Existing frameworks 465 provide comprehensive functional and curated evidence but typically optimise their 466 output for downstream computational analysis or research-oriented exploration, rather 467 than for structured review by multidisciplinary research teams. SMART bridges this gap 468 by consolidating consequence annotation, somatic-focused resources, and OncoKB-469 curated evidence into a single workflow with tiered, audience-targeted outputs. 470 471 A central design feature is the unified transcript prioritisation strategy applied identically 472 at the OncoKB annotation step (Section 4.1) and the table-generation step (Section 473 4.3). As demonstrated by the EGFR L858R and FGFR1 N546K cases (Section 4.5.2), 474 transcript choice can determine whether a variant with therapeutic evidence is 475 recognised or silently lost, and inconsistency between the transcript queried and the 476 transcript displayed is otherwise undetectable in research review. Enforcing a single 477 prioritisation logic across the pipeline removes this class of error. 478 479 In current practice, molecular scientists and bioinformaticians frequently complement 480 automated annotation by manually cross-checking variants of interest across multiple 481 online resources, for example looking up individual alterations in the OncoKB, ClinVar, 482 CIViC, COSMIC, and gnomAD web interfaces, reconciling differences in transcript 483 representation and nomenclature, and recording the consulted database versions for 484 traceability. This process is time-consuming, error-prone, and difficult to standardise 485 across reviewers, particularly when cohorts contain large numbers of variants or when 486

Demonstrated portability across a desktop workstation, a personal 501 laptop, and a high-performance computing cluster (Section 4.6) confirms that SMART 502 can be deployed without specialised infrastructure. 503 504 SMART is particularly well-suited to targeted-panel workflows where the analytical goal 505 is not only broad annotation but also the generation of concise, interpretable outputs for 506 multidisciplinary research discussion.
